Opening an OSGL Window

Introduction

This tutorial will only explain how to open and manage a window. A 'window' is just a fancy name for an EditableImage. Drawing stuff on the window cannot be done with the WindowRenderingContext alone, but requires other OSGL sub-modules that are covered in other tutorials. Either way, window management remains the same so reading this tutorial is important in any case.

Opening a window

In OSGL, Windows can be created by using the OSGL.window class. A Window can be created directly upon construction:
local OSGL = require(script.OSGL) local window = OSGL.window local enum = OSGL.enum local myWindow = window.new({ Size = enum.WindowSize.Maximum, Instance = script.Parent })()
The window constructor, window.new, takes in 1 argument of type: windowParameters. Only 3 possible values can be passed in windowParameters, them being:

  • Instance (GuiObject / EditableImage)
  • Size? (Vector2 / OSGLEnum.OSGLWindowSize)
  • Name? (string)


The Size and Name parameters are both optional and their functionality is pretty obvious. The required Instance parameter however changes how your window will be created.

  • If your Instance is any old GuiObject, then an ImageLabel is created as a child of that Instance, with the EditableImage inside of that.
  • If your Instance is an ImageLabel or ImageButton, then an EditableImage is created as a child of that image object.
  • If your Instance is an EditableImage, that EditableImage will become the window.


The Size parameter must be provided as a Vector2 or an OSGLEnum.OSGLWindowSize. The size is the size of the window in pixels, like a UDim2s offset property. OSGLEnum.OSGLWindowSize has 3 possible values:

  • Maximum
  • Minimum
  • Mutuable


The Maximum and Minimum are just the biggest and smallest sizes a window can be (1024x1024 and 1x1), but a window with the size 'Mutuable' does not have a fixed size. While it is still limited to 1x1 and 1024x1024, the size will be set to the UDim2 of the Instance property, meaning it will be scaled and can be any size inbetween.

If you want to create the window, but aren't planning to use it until a later date, simply don't call the window constructor making it blank data:
local OSGL = require(script.OSGL) local window = OSGL.window local enum = OSGL.enum local myWindow = window.new({ Size = enum.WindowSize.Maximum, Instance = script.Parent })

Bringing our window to life

If you run the window-construction code from earlier, you'll most likely just see the GuiObject you placed before. Why does nothing happen? Firstly, the script ends immediately. Secondly, because we don't render anything and there's no event handling - Even if you added an infinite loop, this would just be a dead window, unable to do be used. Let's add some code to make this program a bit more interesting:
local OSGL = require(script.OSGL) local window = OSGL.window local enum = OSGL.enum local color = OSGL.color local myWindow = window.new({ Size = enum.WindowSize.Maximum, Instance = script.Parent })() while myWindow:isOpen() do myWindow:clear(color.Black) --// ... myWindow:render() end
The above code opens a window, and while it's open, clears the screen with a black color. Let's look at it in detail. The while loop is known as an main loop or game loop - it's the place where we render everything.

We check while the window is "open" (if it exists) using :isOpen(number), which yields for the given amount of time (or just a heartbeat). It returns if the window exists.

A special function called clear is then used - This function clears the argument buffer (I will get into this later) and replaces every pixel with the given color. In this case, every pixel is set to color.Black (0,0,0,0). Finally, these changes are rendered to the screen. After the window is deleted, the program exits.

The argument buffer & drawing stuff

If you've ever done graphics before, you may of heard of a 'double buffer'. For those who don't know, a double buffer is when you write to an array which gets pasted onto the screen meaning we don't need to write to the screen thousands of times - only once. However, while you may think this solution is optimal - it is not.

Roblox's ecosystem already takes up a lot of performance - and creating a massive array and storing RGBA values won't make it any more efficient. If you had a 300x300 window, that's a 360,000 (300x300x4) long array that needs to be stored in memory - and don't forget that we would have to manipulate that too to write other things to it.

To 'solve' this issue (or atleast save on performance), OSGL uses an "argument buffer". The argument buffer stores the necessary 'arguments' and buffers them in memory so they can be rendered at a later date. The argument buffer is always cleared once the ':clear(RGBA)' method is called.

Everything in OSGL uses the argument buffer and nothing is rendered onto the screen instantly. While this may cause minor lag spikes if you're rendering every heartbeat, it is much more efficient than storing a huge array when working with larger images.

The WindowRenderingContext provides 2 functions: clear (as we just covered) and draw. Draw is as simple as its name implies: it draws whatever object you pass to it in the argument buffer.

Calling clear before drawing anything is mandatory, otherwise contents from the previous argument buffer will also be rendered ON TOP of the already rendered entities. Calling render is also mandatory as it takes the argument buffer and gives it to the WindowRenderer, which does exactly what its name says.

Can I draw stuff already??

Now that you understand what a main loop is - you're ready to start drawing on the screen. There are 4 main types of entities that you can draw onto the screen. OSGL provides 5 "ready-made" ones: (sprites & textures, texts and fonts, and shapes), and the other one allows you to custom-render your own primitives through the argument buffer.
